- history of vsc starts with different version of files (filev1.txt,filev1_1.txt), word processors track changes, even undo is a type of version control
- first version control systems - SCCS(1972) -> RCS(1982) -> CVS(1986) -> SVN(2000) -> GIT(2005) -> our friend GitHub(2008)
- in centralized version control system you have centralized server, you need to synchronize to server often and if central server fails developments stops
- in distributed each user maintain their own repositories
- in distributed changes are stored as change sets
- in distributed you track changes, not versions
- in distributed sets can be exchanged between repositories, not only with the centralized server
- you might have many repos with different sets
- distributed is: faster, no network access required, no single failure point, encourages participation, more convenient for open source communities
- in distributed you can still designate primary repository for collaboration
- review the terms and commands below to get an idea of how Git is structured
- in Git there are four types of objects : `blob, tree, commit, tag`
- SHA for identical files is the same regardless of their location at the given moment
- when you initialize your empty repo take a look what is inside of .git dir `find .` to list all files
- all versions of HEAD are -> HEAD, ORIGIN_HEAD, MERGE_HEAD, FETCH_HEAD => command to manage is `git symbolic-ref HEAD` - deals with named references (branches, HEAD)
- `git rev-parse <name>` resolves anything - a name, tag, reference, result is sha
